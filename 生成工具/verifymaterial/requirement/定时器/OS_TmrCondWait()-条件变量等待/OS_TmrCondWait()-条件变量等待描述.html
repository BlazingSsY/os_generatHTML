需求描述

需求编号：HLR_OS_06_16

工作描述：HLR：uC/OS-Ⅲ应提供条件变量等待的功能。

是否需求：是

验证方法：测试验证

补充信息：

1.函数名称

OS_OS_TmrCondWait()

OS_TmrCondWait() --- 条件变量等待

2.函数框架

static void OS_TmrCondWait (
    OS_TICK timeout /* 输入：等待超时时间 */
)

3.函数描述

该函数用于在条件变量上等待。定时器任务会释放互斥信号量，并进入等待状态，直到条件变量被信号触发或者等待超时。这个函数主要用于同步机制，确保定时器任务在特定条件下才继续执行。

4.参数说明

（1）timeout

类型：OS_TICK

作用：等待超时时间（时钟节拍数）。

取值：

0u：无限等待，直到条件变量被信号唤醒。

>0u：最多等待指定节拍数，超时后返回

5.算法逻辑

1）释放互斥信号量

  1. 调用 CPU_CRITICAL_ENTER() 进入临界区，防止在执行关键操作时被中断干扰。
  2. 调用 OS_MutexGrpRemove() 将互斥信号量 OSTmrMutex 从定时器任务的互斥信号量组中移除，释放互斥信号量以便其他任务可以获取。

2）恢复任务优先级

如果定时器任务的当前优先级不等于其基础优先级，说明之前发生了优先级继承。此时需要恢复定时器任务的原始优先级，并相应地更新系统的当前优先级（OSPrioCur）。

3）检查等待列表并处理

  1. 获取互斥信号量的等待列表。
  2. 如果等待列表为空，则将互斥信号量的所有者指针和嵌套计数设置为 NULL 和 0。
  3. 如果等待列表不为空，则将等待列表中第一个任务设置为互斥信号量的新所有者，并更新相关指针和计数。调用 OS_Post() 函数向互斥信号量发送信号，唤醒等待任务。

4）在条件变量上等待

调用 OS_Pend() 函数，使定时器任务在条件变量 OSTmrCond 上等待，等待超时时间为 输入参数timeout。

5）退出临界区，并开启任务调度

调用 CPU_CRITICAL_EXIT() 退出临界区，允许中断继续发生，调用 OSSched() 函数进行任务调度。

6）再次进入临界区，并尝试获取互斥信号量

再次调用 CPU_CRITICAL_ENTER() 进入临界区，处理等待结束后的状态。

  1. 如果互斥信号量的所有者指针为空，说明可以获取互斥信号量。将定时器任务添加到互斥信号量组，并设置互斥信号量的所有者指针和嵌套计数。退出临界区。
  2. 如果互斥信号量的所有者指针不为空，则需要等待获取互斥信号量。获取当前所有者任务，如果所有者任务的优先级低于定时器任务的优先级，则提升所有者任务的优先级。调用 OS_Pend() 函数使定时器任务在互斥信号量上等待，然后退出临界区并进行任务调度。

6.返回值

无。

7.健康管理错误码

无。

备注：此函数是uC/OS-III的内部函数，您的应用程序不得调用它。
